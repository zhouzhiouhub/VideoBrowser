## 1. 文档用途

本文件作为后续开发入口。每次开始开发前，先查看本文件的“当前开发指针”和“阶段任务表”，再按顺序推进。

`开发设计文档.md` 负责说明产品目标、架构和技术方案；本文件负责说明开发顺序、完成度、验收标准和下一步任务。

## 2. 每次开发流程

每次开发必须按以下流程执行：

1. 阅读“当前开发指针”，确认本次应该做哪一个阶段、哪一个任务。
2. 阅读对应阶段的“进入条件”和“验收标准”。
3. 先确认本次代码应归属的模块和包名，禁止把新功能继续堆入单个 Activity 或单个大文件。
4. 只实现当前任务以及它直接依赖的最小改动。
5. 完成后执行对应验证命令或手工验证。
6. 更新本文件中的任务状态、阶段完成度、验证记录和下一步任务。
7. 明确提醒开发工程师进行测试，列出建议测试项。
8. 询问开发工程师是否继续进行下一步开发。
9. 未获得开发工程师确认前，不进入下一任务或下一阶段。
10. 未通过验收时，不进入下一阶段。

代码规范与模块化要求：

- `MainActivity` 只负责页面控件绑定、用户事件分发、Activity 生命周期转发和轻量 UI 状态更新。
- `WebView` 初始化、浏览器导航、加载状态和生命周期封装到 `browser/`。
- URL、域名、搜索词解析封装到 `utils/`，不要散落在 Activity 内。
- 广告请求拦截封装到 `adblock/`；规则数据结构和匹配逻辑封装到 `rules/`。
- JS 脚本加载和注入封装到 `inject/`；站点识别和站点脚本选择封装到 `site/`。
- 设置读取、默认值和持久化封装到 `settings/`、`storage/`，不要由 UI 层直接承担业务规则。
- 每次新增功能时优先创建或扩展对应功能模块，只有 UI 绑定代码可以留在 `MainActivity`。
- 修改已有大文件时，若本次任务涉及独立业务逻辑，应优先迁移到对应模块，再在 UI 层调用。
- 新文件命名应表达单一职责，避免 `Manager` 文件继续膨胀成跨领域集合；跨阶段功能不得混放在同一类中。
- 阶段完成前必须检查改动范围是否符合模块边界，并在验证记录或备注中说明模块化处理结果。

强制停点：

- 每完成一个任务，都必须停止自动推进。
- 必须先提醒开发工程师测试当前任务。
- 必须询问开发工程师是否进行下一步。
- 只有开发工程师明确回复继续后，才允许进入下一任务。

## 3. 状态标记规则

任务状态使用以下格式：

| 标记 | 含义 |
| --- | --- |
| `[ ]` | 未开始 |
| `[~]` | 进行中 |
| `[x]` | 已完成 |
| `[!]` | 阻塞或存在风险 |
| `[-]` | 延后，不属于当前阶段 |

完成度填写规则：

- 0%：未开始。
- 25%：已完成基础结构，但不可完整验证。
- 50%：核心功能已实现，可局部验证。
- 75%：主要验收项通过，仍有边界问题或测试缺口。
- 100%：阶段验收项全部通过，并已更新验证记录。
