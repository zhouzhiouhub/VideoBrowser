## 1. 文档用途

本文件作为后续开发入口。每次开始开发前，先查看本文件的“当前开发指针”和“阶段任务表”，再按顺序推进。

`开发设计文档.md` 负责说明产品目标、架构和技术方案；本文件负责说明开发顺序、完成度、验收标准和下一步任务。

## 2. 每次开发流程

每次开发必须按以下流程执行：

1. 阅读“当前开发指针”，确认本次应该做哪一个阶段、哪一个任务。
2. 阅读对应阶段的“进入条件”和“验收标准”。
3. 先确认本次代码应归属的模块和包名，禁止把新功能继续堆入单个 Activity 或单个大文件。
4. 只实现当前任务以及它直接依赖的最小改动。
5. 完成后执行对应验证命令或手工验证。
6. 更新本文件中的任务状态、阶段完成度、验证记录和下一步任务。
7. 明确提醒开发工程师进行测试，列出建议测试项。
8. 询问开发工程师是否继续进行下一步开发。
9. 未获得开发工程师确认前，不进入下一任务或下一阶段。
10. 未通过验收时，不进入下一阶段。
11. 修改代码时必须同步补充中文注释，便于开发人员快速判断本次新增或调整了哪些逻辑。

代码规范与模块化要求：

- `MainActivity` 只负责页面控件绑定、用户事件分发、Activity 生命周期转发和轻量 UI 状态更新。
- `WebView` 初始化、浏览器导航、加载状态和生命周期封装到 `browser/`。
- URL、域名、搜索词解析封装到 `utils/`，不要散落在 Activity 内。
- 广告请求拦截封装到 `adblock/`；规则数据结构和匹配逻辑封装到 `rules/`。
- JS 脚本加载和注入封装到 `inject/`；站点识别和站点脚本选择封装到 `site/`。
- 设置读取、默认值和持久化封装到 `settings/`、`storage/`，不要由 UI 层直接承担业务规则。
- 每次新增功能时优先创建或扩展对应功能模块，只有 UI 绑定代码可以留在 `MainActivity`。
- 修改已有大文件时，若本次任务涉及独立业务逻辑，应优先迁移到对应模块，再在 UI 层调用。
- 新文件命名应表达单一职责，避免 `Manager` 文件继续膨胀成跨领域集合；跨阶段功能不得混放在同一类中。
- 新增或修改代码时必须添加中文注释；注释应说明新增逻辑的目的、关键业务边界、兼容性处理或非显然判断，避免只重复代码字面含义。
- 阶段完成前必须检查改动范围是否符合模块边界，并在验证记录或备注中说明模块化处理结果。

原生播放器 UI 复用规则（重要）：

- 如果播放器、系统或组件库已经提供现成功能，必须优先调用现有 API、复用原生控件和监听已有状态，只补充缺失能力。
- 手势层只负责控制播放器行为、调用播放器 API 和提供必要 HUD；不得重绘播放器核心 UI，也不得替代原生播放器控件。
- 禁止重复实现播放 / 暂停图标、视频进度条、缓冲进度、Loading 动画、音量系统、亮度层、全屏切换、倍速逻辑、视频时间状态和字幕渲染。
- 播放 / 暂停只能调用播放器 `play()` / `pause()`，状态展示交给原生播放器 UI；不得自定义中央播放按钮或暂停动画。
- 视频进度必须使用播放器原生 seek bar；仅允许增强拖动交互、横向滑动 seek HUD 和快进快退 HUD。
- 缓冲、Loading、时长、当前时间和字幕必须读取播放器原生状态，不允许另维护一套状态或绘制一层同类 UI。
- 音量和亮度必须调用系统能力；允许自定义百分比 HUD，但不允许做假的音量系统或亮度遮罩层。
- 全屏必须使用播放器原生 fullscreen API 或系统方向控制；不得硬改 View Rotation 形成第二套横竖屏状态。
- 倍速必须调用播放器 `playbackSpeed` / `playbackRate`；允许自定义倍速菜单 UI。
- 允许自定义手势识别、HUD 反馈、锁屏按钮、横向滑动 Seek HUD、音量 HUD、亮度 HUD、快进快退动画、倍速菜单 UI、横竖屏按钮 UI。
- 正确架构是 `Gesture Layer -> Player Controller -> Native Player UI`，不是 `Gesture Layer -> Custom UI Overlay -> Native UI`。
- 视频增强目标是增强播放器交互，不是重做播放器；后续不得继续新增或扩展重复的 HTML5 视频浮层控制条。
- 开发前必须检查目标能力是否已有原生实现；完成后在验证记录或备注中说明复用的原生 API、原生控件或必须补充的缺失能力。

强制停点：

- 每完成一个任务，都必须停止自动推进。
- 必须先提醒开发工程师测试当前任务。
- 必须询问开发工程师是否进行下一步。
- 只有开发工程师明确回复继续后，才允许进入下一任务。

## 3. 状态标记规则

任务状态使用以下格式：

| 标记 | 含义 |
| --- | --- |
| `[ ]` | 未开始 |
| `[~]` | 进行中 |
| `[x]` | 已完成 |
| `[!]` | 阻塞或存在风险 |
| `[-]` | 延后，不属于当前阶段 |

完成度填写规则：

- 0%：未开始。
- 25%：已完成基础结构，但不可完整验证。
- 50%：核心功能已实现，可局部验证。
- 75%：主要验收项通过，仍有边界问题或测试缺口。
- 100%：阶段验收项全部通过，并已更新验证记录。
